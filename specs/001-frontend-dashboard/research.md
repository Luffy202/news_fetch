# Research: 微信公众号前端管理看板

## 1. Backend API service pattern

**Decision**

采用轻量级 `FastAPI` 服务封装现有同步爬虫模块，并通过 REST API 提供前端所需的任务触发、进度查询、历史记录、设置管理和飞书推送能力。

**Rationale**

- 现有代码已经按模块分离为 `fetcher.py`、`parser.py`、`notifier.py`、`summarizer.py`、`auth.py`，适合在不大改内部逻辑的前提下增加一层 API 封装。
- `FastAPI` 可以直接暴露同步函数风格的路由，减少对现有爬虫逻辑的重构成本。
- 规划中的接口包括：启动爬取、获取当前任务状态、查询历史批次、更新设置、触发飞书推送、触发登录/查看登录状态，均适合用 HTTP API 表达。
- WeChat 扫码登录仍在服务端通过 Playwright 完成，前端只负责发起登录和查看状态。
- 单活任务约束可由进程内任务管理器统一控制，避免并发爬取导致限流和状态混乱。

**Alternatives considered**

- Flask：可行，但内建的接口校验、文档生成、后台任务配合度不如 `FastAPI`，需要更多手工样板代码。
- 直接保留 CLI、由前端调用 shell：耦合高，状态管理和错误回传复杂，不利于历史记录与看板实现。

## 2. Storage and persistence model

**Decision**

采用 `SQLite` 作为主持久化存储，必要时辅以导出用 Markdown/ZIP 文件，不再依赖单一 JSON 文件作为唯一数据源。

**Rationale**

- 该项目是单用户、本机或受控内网场景，`SQLite` 足够满足账号列表、设置项、历史批次、文章内容和任务状态的持久化需求。
- 相比多个 JSON 文件，`SQLite` 更适合保存批次与文章的关联关系，便于做历史查询、统计看板和导出。
- 现有 `output/articles.json` 可继续作为兼容性输出或调试产物，但不应作为前端状态和历史统计的唯一来源。
- Markdown/ZIP 下载属于导出能力，应以数据库中的文章数据为基础生成，而不是反向解析已有输出文件。

**Alternatives considered**

- 多 JSON 文件：实现简单，但历史批次关联、统计查询、任务状态一致性较差。
- 更重的数据库服务：超出单用户本地工具需要，增加部署与维护成本。

## 3. Frontend interaction pattern

**Decision**

前端采用单页管理台模式：左侧/顶部用于公众号列表与操作控制，中部展示当前任务进度与结果，辅以历史批次列表、文章详情面板、数据看板页面和设置页面。

**Rationale**

- 用户的核心流程是：管理公众号 → 选择公众号 → 发起一轮爬取 → 查看当前结果 → 推送当前轮次到飞书 → 查看历史/导出。
- 当前任务与历史批次都很重要，因此 UI 需要同时表达“当前正在发生什么”和“过去发生了什么”。
- 单用户场景下，无需复杂权限分层，可将主要流程集中在少量页面中完成，提高操作效率。
- Markdown 导出、飞书推送都应基于“当前轮次”或“历史批次”的明确上下文，避免作用范围不清。

**Alternatives considered**

- 多层级复杂后台导航：信息架构过重，不适合单用户工具。
- 仅保留当前结果、不保留历史：无法支撑数据看板和重复查看需求。

## 4. QR login handling

**Decision**

保留现有 Playwright 扫码登录机制，由后端 API 触发登录流程并暴露登录状态；前端负责提示用户开始登录、轮询状态，并在需要时引导重新认证。

**Rationale**

- 现有 `auth.py` 已能完成扫码登录与凭证提取，复用成本最低。
- 扫码行为本质上发生在服务端打开的浏览器环境中，不适合前端直接接管。
- 前端只需表达三种状态：未登录、登录中、登录成功/失败，并在 session 失效时引导重新登录。

**Alternatives considered**

- 前端直接嵌入扫码登录流程：与现有实现不匹配，需要重做认证链路。
- 仅手动编辑配置文件：不适合前端管理台体验。

## 5. Single active crawl enforcement

**Decision**

同一时间只允许一轮爬取任务处于进行中状态；当已有任务执行中时，新请求必须被阻止或置为等待，并在前端明确提示。

**Rationale**

- 现有爬虫具备显式速率限制，说明上游平台对访问频率敏感，并发任务会放大失败率和状态复杂度。
- “当前这一轮”的飞书推送范围、历史记录归属和进度展示都建立在单轮任务边界清晰的前提下。
- 单活任务实现更简单、更符合当前项目体量，也能降低后续排障成本。

**Alternatives considered**

- 多任务并行：会增加限流风险，并使轮次边界、任务状态和推送范围变复杂。
- 无限排队：虽可行，但首版不必引入完整调度器；允许“阻止或等待”即可满足规格。
