# Feature Specification: 微信公众号前端管理看板

**Feature Branch**: `001-frontend-dashboard`
**Created**: 2026-03-13
**Status**: Draft
**Input**: User description: "为微信公众号爬取工具创建前端管理界面，使用Vite技术栈，支持公众号管理、文章爬取、飞书推送、源数据看板和Markdown下载"

## Clarifications

### Session 2026-03-13

- Q: 该前端面向什么用户范围？→ A: 单用户使用，不做登录，仅限本机或内网受限访问
- Q: 爬取结果保留策略是什么？→ A: 保留每次爬取历史，可反复查看
- Q: 飞书推送默认推送什么范围？→ A: 推送当前这一轮中用户选中公众号的爬取结果
- Q: 是否允许同时发起多轮爬取？→ A: 同一时间只允许一轮爬取，新的请求需等待或被阻止

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 公众号管理 (Priority: P1)

用户打开前端管理界面后，可以在公众号管理页面中添加、编辑和删除要关注的微信公众号。用户输入公众号名称后，系统将其保存到关注列表中。用户可以在列表中查看所有已添加的公众号，并通过勾选框选中一个或多个公众号作为本次爬取目标。

**Why this priority**: 公众号管理是整个工具的基础操作入口。没有公众号列表，后续的爬取、推送等功能均无法执行。

**Independent Test**: 可以通过添加3个公众号、编辑其中1个名称、删除1个、选中剩余2个来独立验证此功能。

**Acceptance Scenarios**:

1. **Given** 用户首次打开管理界面, **When** 用户在输入框中输入公众号名称并点击"添加", **Then** 该公众号出现在关注列表中，并默认处于未选中状态
2. **Given** 列表中已有多个公众号, **When** 用户勾选其中若干个, **Then** 被勾选的公众号高亮显示，系统记录选中状态
3. **Given** 列表中已有某公众号, **When** 用户点击"删除"按钮, **Then** 该公众号从列表中移除
4. **Given** 用户刷新页面, **When** 页面重新加载, **Then** 之前添加的公众号列表和选中状态保持不变

---

### User Story 2 - 文章爬取与结果查看 (Priority: P1)

用户选中目标公众号后，点击"开始爬取"按钮，系统调用后端爬取服务获取文章数据。爬取过程中，界面显示进度状态（如"正在爬取 XX 公众号..."）。爬取完成后，用户可以在文章列表中查看每篇文章的标题、摘要、发布时间和正文内容。

**Why this priority**: 文章爬取是核心业务功能，是用户使用此工具的主要目的。

**Independent Test**: 可以通过选中1个公众号、触发爬取、查看返回的文章列表来独立验证。

**Acceptance Scenarios**:

1. **Given** 用户已选中至少1个公众号, **When** 用户点击"开始爬取", **Then** 系统显示爬取进度，并在完成后展示文章列表
2. **Given** 爬取正在进行中, **When** 用户查看界面, **Then** 界面显示当前正在处理的公众号名称和进度信息
3. **Given** 爬取完成, **When** 用户点击某篇文章标题, **Then** 可以查看该文章的完整正文内容
4. **Given** 未选中任何公众号, **When** 用户点击"开始爬取", **Then** 系统提示"请先选择至少一个公众号"

---

### User Story 3 - 飞书推送 (Priority: P2)

用户完成一轮爬取后，可以将当前这一轮中用户选中公众号的爬取结果推送到飞书群。用户可以在设置中配置飞书 Webhook 地址。推送完成后，界面显示推送成功或失败的状态反馈。

**Why this priority**: 飞书推送是内容分发的重要渠道，但依赖于爬取功能先完成，属于增值功能。

**Independent Test**: 可以通过配置 Webhook 地址、完成一轮针对选中公众号的爬取、触发推送、检查飞书群收到消息来独立验证。

**Acceptance Scenarios**:

1. **Given** 用户已完成当前一轮爬取且已配置飞书 Webhook, **When** 用户点击"推送到飞书", **Then** 当前这一轮中用户选中公众号的爬取结果以卡片形式推送到飞书群，界面显示"推送成功"
2. **Given** 飞书 Webhook 未配置, **When** 用户尝试推送, **Then** 系统提示"请先在设置中配置飞书 Webhook 地址"
3. **Given** 推送过程中网络异常, **When** 推送失败, **Then** 界面显示错误提示，用户可以重试

---

### User Story 4 - 源数据看板 (Priority: P2)

用户可以在数据看板页面查看爬取数据的统计概览，包括：已关注公众号数量、已爬取文章总数、各公众号文章数量分布、最近爬取时间等。看板以可视化图表和统计卡片的形式呈现关键数据指标。

**Why this priority**: 数据看板帮助用户了解整体数据情况，提升工具的可用性和专业感，但不影响核心爬取流程。

**Independent Test**: 可以通过爬取若干公众号文章后查看看板页面，验证统计数据是否准确来独立测试。

**Acceptance Scenarios**:

1. **Given** 用户已爬取过多个公众号的文章, **When** 用户打开数据看板页面, **Then** 看板显示公众号总数、文章总数、各公众号文章数量分布图
2. **Given** 尚未进行任何爬取, **When** 用户打开数据看板, **Then** 看板显示空状态提示，引导用户去添加公众号并开始爬取
3. **Given** 新一轮爬取完成, **When** 用户查看看板, **Then** 统计数据自动更新，反映最新爬取结果

---

### User Story 5 - Markdown 下载 (Priority: P3)

用户可以将爬取到的文章内容导出为 Markdown 格式文件并下载到本地。用户可以选择单篇文章下载，也可以批量选择多篇文章打包下载。Markdown 文件包含文章标题、发布时间、来源公众号和正文内容。

**Why this priority**: Markdown 下载是内容沉淀和二次利用的便捷方式，但属于锦上添花的辅助功能。

**Independent Test**: 可以通过选中已爬取的文章、点击下载、检查下载的 .md 文件内容格式来独立验证。

**Acceptance Scenarios**:

1. **Given** 用户已爬取到文章, **When** 用户选中一篇文章并点击"下载为 Markdown", **Then** 浏览器下载一个 .md 文件，文件包含标题、时间、来源和正文
2. **Given** 用户选中多篇文章, **When** 用户点击"批量下载", **Then** 系统将多篇文章打包为 .zip 文件下载，每篇文章一个 .md 文件
3. **Given** 文章正文中包含特殊字符, **When** 导出为 Markdown, **Then** 特殊字符被正确转义，Markdown 格式可正常渲染

---

### Edge Cases

- 用户添加了重复名称的公众号时，系统应提示"该公众号已存在"并阻止重复添加
- 爬取过程中 WeChat 后台 session 过期（约2小时），系统应提示用户重新认证
- 飞书推送内容过长时（超过飞书消息卡片限制），系统应自动截断并附上"查看更多"提示
- 同时爬取大量公众号时，后端需按顺序处理并遵守速率限制，前端应展示排队状态
- 当已有一轮爬取任务进行中时，用户再次点击开始爬取，系统应阻止或等待新请求，并明确提示当前仅支持单轮任务运行
- 用户在爬取进行中关闭浏览器后重新打开，应能查看上次爬取的历史结果
- 下载 Markdown 时文章正文为空（抓取失败），文件中应标注"[正文内容获取失败]"

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须提供公众号管理功能，支持添加、删除和选中/取消选中公众号
- **FR-001A**: 系统必须按单用户场景设计，不要求提供登录、注册或角色权限管理能力，部署范围限定为本机或受控内网环境
- **FR-002**: 系统必须持久化存储公众号列表和选中状态，刷新页面后数据不丢失
- **FR-002A**: 系统必须持久化保存每次爬取历史结果，并支持用户反复查看历史批次的文章内容、统计数据和导出内容
- **FR-003**: 系统必须防止添加重复名称的公众号
- **FR-004**: 系统必须提供文章爬取触发功能，对选中的公众号批量执行爬取操作
- **FR-005**: 系统必须在爬取过程中显示实时进度反馈，包括当前处理的公众号和完成百分比
- **FR-006**: 系统必须展示爬取到的文章列表，包含标题、摘要、发布时间等基本信息
- **FR-007**: 系统必须支持查看单篇文章的完整正文内容
- **FR-008**: 系统必须支持配置飞书 Webhook 地址，并将配置持久化保存
- **FR-009**: 系统必须支持将当前这一轮中用户选中公众号的爬取结果推送至配置的飞书群，并显示推送结果状态
- **FR-010**: 系统必须提供数据看板页面，展示公众号数量、文章总数、各公众号文章分布等统计信息
- **FR-011**: 系统必须支持将单篇文章导出为 Markdown 格式文件并下载
- **FR-012**: 系统必须支持批量选择多篇文章并打包下载为 .zip 文件
- **FR-013**: 导出的 Markdown 文件必须包含文章标题、发布时间、来源公众号和正文内容
- **FR-014**: 系统必须提供后端 API 服务，供前端界面调用，封装现有的爬取、推送等能力
- **FR-015**: 系统必须处理 WeChat session 过期场景，在凭据失效时提示用户重新认证
- **FR-016**: 系统必须限制同一时间仅能存在一轮进行中的爬取任务；当已有任务执行中时，新的爬取请求必须被阻止或进入等待状态，并向用户明确反馈当前状态

### Key Entities

- **公众号 (Account)**: 代表一个微信公众号，包含名称、fakeid（平台唯一标识）、添加时间、选中状态
- **文章 (Article)**: 代表一篇公众号文章，包含标题、摘要、正文内容、发布时间、来源公众号、文章URL、封面图URL
- **爬取任务 (Crawl Task)**: 代表一次爬取操作，包含目标公众号列表、开始时间、状态（等待中/进行中/已完成/失败）、结果统计，并作为可重复查看的历史批次保留；同一时间只允许一条任务处于进行中状态
- **系统配置 (Settings)**: 包含飞书 Webhook 地址、爬取文章数量、请求间隔等可配置参数

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户从打开界面到完成首次公众号添加操作不超过30秒
- **SC-002**: 用户触发爬取后，界面在2秒内显示爬取进度反馈
- **SC-003**: 单次爬取10篇文章的结果能在界面上完整展示，包含标题、摘要和正文
- **SC-004**: 飞书推送操作完成后，用户在5秒内看到成功或失败的状态反馈
- **SC-005**: 数据看板在页面加载后2秒内展示所有统计图表
- **SC-006**: Markdown 文件下载在用户点击后3秒内开始
- **SC-007**: 批量下载20篇文章的 .zip 文件大小合理（文本内容为主，单文件不超过5MB）
- **SC-008**: 页面刷新后，所有用户数据（公众号列表、配置、历史爬取结果）完整保留
- **SC-009**: 用户可以在不查阅文档的情况下完成所有主要操作（添加公众号、爬取、推送、下载）

## Assumptions

- 前端通过 Vite 构建，通过 HTTP API 与后端 Python 服务通信
- 后端需要新增一个 API 层来暴露现有的爬取、推送功能
- 公众号列表和爬取结果持久化存储在后端（文件或轻量级数据库），前端通过 API 读写
- 飞书推送复用现有 notifier.py 的逻辑，通过 API 端点触发
- WeChat 认证流程仍需用户手动扫码，前端提供引导入口但实际扫码在服务端浏览器窗口完成
- 数据看板的统计基于已爬取并存储的历史数据，不需要实时连接 WeChat 平台
- Markdown 导出在前端完成（基于已获取的文章数据），不需要额外的后端支持
